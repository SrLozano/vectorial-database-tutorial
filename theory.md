## 1. 🗄️ Vector Databases

### What is a Vector Database?

A **vector database** is a specialized database designed to store, index, and query high-dimensional vector embeddings efficiently. Unlike traditional databases that store structured data (rows and columns), vector databases are optimized for similarity search operations.

### Key Characteristics

- **High-dimensional data**: Vectors typically have hundreds or thousands of dimensions
- **Similarity search**: Find items similar to a query vector, not exact matches
- **Scalability**: Handle billions of vectors with millisecond latency
- **Metadata support**: Combine vector search with traditional filtering

### Why Milvus?

Milvus is an open-source vector database that provides:
- **Multiple index types**: IVF_FLAT, HNSW, IVF_SQ8, etc.
- **Hybrid search**: Combine dense and sparse vectors
- **Distributed architecture**: Scale horizontally
- **Rich ecosystem**: Integration with ML frameworks

### Use Cases

1. **Semantic Search**: Find documents by meaning, not just keywords
2. **Recommendation Systems**: Suggest similar items based on user preferences
3. **Image/Video Search**: Find visually similar content
4. **Question Answering**: Retrieve relevant context for LLMs (RAG)
5. **Anomaly Detection**: Identify outliers in high-dimensional data


## 2. 🧬 Vector Embeddings

### What are Embeddings?

**Embeddings** are numerical representations (vectors) of data that capture semantic meaning in a continuous vector space. They transform discrete objects (words, sentences, images) into points in a multi-dimensional space where similar items are close together.

### Types of Embeddings

#### Dense Embeddings

Dense embeddings are continuous vectors where most values are non-zero. They capture semantic meaning and context.

**Characteristics:**
- Fixed dimensionality (e.g., 768, 1024, 1536)
- All dimensions have values
- Capture semantic relationships
- Generated by neural networks

**Example Models:**
- **BERT**: Bidirectional Encoder Representations from Transformers
- **Sentence-BERT**: Optimized for sentence-level embeddings
- **E5**: Multilingual embedding models
- **OpenAI Ada**: High-quality general-purpose embeddings

**Example:**
```python
# Text: "The cat sat on the mat"
# Dense embedding (simplified, actual is 1024-dim):
[0.23, -0.45, 0.67, 0.12, -0.89, ...]
```

#### Sparse Embeddings

Sparse embeddings are vectors where most values are zero. They're used for lexical/keyword-based search.

**Characteristics:**
- Variable dimensionality
- Most values are zero (sparse)
- Capture exact term matches
- Generated by statistical methods (BM25, TF-IDF)

**Example:**
```python
# Text: "vector database search"
# Sparse embedding (only non-zero indices shown):
{45: 2.3, 127: 1.8, 892: 3.1}  # index: weight
```

### How Embeddings Work

1. **Training**: Neural networks learn to map inputs to vectors
2. **Semantic Space**: Similar concepts cluster together
3. **Distance**: Proximity indicates similarity

**Example Semantic Space:**
```
"king" - "man" + "woman" ≈ "queen"
```

### Embedding Generation in the Tutorial

The tutorial uses an embedding service to generate dense embeddings:

```python
# From embeddings_utils.py
embedding = EmbeddingModel(
    model_id="intfloat/multilingual-e5-large",  # 1024 dimensions
    project_id=self.project_id,
    params=embed_params,
    client=self.client,

embedding_vectors = embedding.embed_documents(texts=text_contents)
```

**Key Parameters:**
- `TRUNCATE_INPUT_TOKENS: 512`: Limit input length
- `RETURN_OPTIONS`: Include metadata in response
- Model outputs 1024-dimensional vectors


## 3. 📏 Similarity Metrics

Similarity metrics measure how close two vectors are in the embedding space. Different metrics are suitable for different use cases.

### Cosine Similarity

**Formula:**
```
cosine_similarity(A, B) = (A · B) / (||A|| × ||B||)
```

**Range:** -1 to 1 (1 = identical direction, 0 = orthogonal, -1 = opposite)

**When to Use:**
- Text embeddings (most common)
- When magnitude doesn't matter, only direction
- Normalized vectors

**In Milvus:**
```python
search_params = {"metric_type": "COSINE"}
```

**Advantages:**
- Invariant to vector magnitude
- Intuitive interpretation
- Works well with normalized embeddings

### Euclidean Distance (L2)

**Formula:**
```
L2_distance(A, B) = √(Σ(Aᵢ - Bᵢ)²)
```

**Range:** 0 to ∞ (0 = identical, larger = more different)

**When to Use:**
- When absolute distances matter
- Image embeddings
- Spatial data

**In Milvus:**
```python
search_params = {"metric_type": "L2"}
```

### Inner Product (IP)

**Formula:**
```
IP(A, B) = A · B = Σ(Aᵢ × Bᵢ)
```

**Range:** -∞ to ∞ (higher = more similar)

**When to Use:**
- Normalized vectors (equivalent to cosine)
- Sparse vectors (BM25)
- Maximum Inner Product Search (MIPS)

**In Milvus:**
```python
search_params = {"metric_type": "IP"}
```

**Note:** For sparse vectors (BM25), IP is the standard metric.

### Choosing the Right Metric

| Metric | Best For | Normalized Vectors | Sparse Vectors |
|--------|----------|-------------------|----------------|
| COSINE | Text embeddings | Yes | No |
| L2 | Image embeddings | No | No |
| IP | Sparse/BM25 | Yes | Yes |


## 4. 🗂️ Index Types

Indexes are data structures that enable fast similarity search. Without indexes, searching requires comparing the query to every vector (brute force), which is slow for large datasets.

### Why Indexes Matter

**Without Index (FLAT):**
- Time complexity: O(n) - linear scan
- Exact results
- Slow for millions of vectors

**With Index:**
- Time complexity: O(log n) or better
- Approximate results (with high accuracy)
- Fast even for billions of vectors

### IVF_FLAT (Inverted File with Flat Compression)

**How it Works:**
1. Cluster vectors into groups (cells) using k-means
2. Store cluster centroids
3. During search, find nearest centroids
4. Search only within those clusters

**Characteristics:**
- **Speed**: Moderate (good balance)
- **Memory**: Moderate
- **Accuracy**: High (configurable)
- **Best for**: General-purpose applications

**Parameters:**
```python
index_params = {
    "index_type": "IVF_FLAT",
    "metric_type": "COSINE",
    "params": {"nlist": 128}  # Number of clusters
}
```

**Trade-offs:**
- `nlist` (clusters): More = slower build, faster search
- `nprobe` (search clusters): More = slower search, higher accuracy

### HNSW (Hierarchical Navigable Small World)

**How it Works:**
1. Build a multi-layer graph structure
2. Each layer is a navigable small-world graph
3. Search starts at top layer, moves down
4. Greedy search within each layer

**Characteristics:**
- **Speed**: Very fast (best search performance)
- **Memory**: High (stores graph structure)
- **Accuracy**: Very high
- **Best for**: Low-latency applications

**Parameters:**
```python
index_params = {
    "index_type": "HNSW",
    "metric_type": "COSINE",
    "params": {
        "M": 16,        # Max connections per node
        "efConstruction": 200  # Build-time search depth
    }
}
```

**Trade-offs:**
- Higher memory usage
- Slower index building
- Fastest search queries

### IVF_SQ8 (IVF with Scalar Quantization)

**How it Works:**
1. Same as IVF_FLAT (clustering)
2. Compress vectors using 8-bit quantization
3. Reduces memory by ~75%

**Characteristics:**
- **Speed**: Fast
- **Memory**: Low (compressed)
- **Accuracy**: Good (slight loss from compression)
- **Best for**: Large-scale deployments with memory constraints

**Parameters:**
```python
index_params = {
    "index_type": "IVF_SQ8",
    "metric_type": "COSINE",
    "params": {"nlist": 128}
}
```

### SPARSE_INVERTED_INDEX

**How it Works:**
1. Inverted index for sparse vectors
2. Maps terms to document IDs
3. Optimized for BM25 sparse embeddings

**Characteristics:**
- **Speed**: Very fast for sparse vectors
- **Memory**: Low (only non-zero values)
- **Accuracy**: Exact
- **Best for**: Lexical/keyword search

**Parameters:**
```python
index_params = {
    "index_type": "SPARSE_INVERTED_INDEX",
    "metric_type": "IP"
}
```

### Index Comparison

| Index Type | Speed | Memory | Accuracy | Use Case |
|------------|-------|--------|----------|----------|
| FLAT | Slow | Low | 100% | Small datasets, exact search |
| IVF_FLAT | Medium | Medium | 95-99% | General purpose |
| HNSW | Fast | High | 98-99% | Low latency required |
| IVF_SQ8 | Fast | Low | 90-95% | Large scale, memory limited |
| SPARSE_INVERTED | Fast | Low | 100% | Lexical search (BM25) |


## 5. 🔍 Search Strategies

### Semantic Search (Dense Vector Search)

**Concept:**
Search based on meaning and context, not exact keywords. Uses dense embeddings to find semantically similar content.

**How it Works:**
1. Convert query text to dense embedding
2. Compare query vector to all document vectors
3. Return top-k most similar documents

**Example:**
```python
Query: "How do I build a recommendation system?"

# Finds documents about:
- Recommendation engines
- Collaborative filtering
- Vector similarity for suggestions
# Even if they don't contain exact words "build" or "recommendation system"
```

**Implementation:**
```python
results = client.semantic_search(
    collection_name=DEMO_COLLECTION,
    query=query,
    anns_field="vector_dense",  # Dense embedding field
    output_fields=["text", "category"],
    limit=5
)
```

**Advantages:**
- Understands synonyms and context
- Finds conceptually related content
- Language-agnostic (with multilingual models)

**Disadvantages:**
- May miss exact keyword matches
- Requires quality embedding models
- Computationally expensive

### Lexical Search (Sparse Vector Search)

**Concept:**
Traditional keyword-based search using statistical methods like BM25. Uses sparse embeddings to find documents with matching terms.

**How it Works:**
1. Convert query to sparse embedding (BM25)
2. Match terms using inverted index
3. Rank by term frequency and document frequency

**Example:**
```python
Query: "vector database indexing"

# Finds documents containing:
- "vector" AND/OR "database" AND/OR "indexing"
# Exact term matches, ranked by relevance
```

**Implementation:**
```python
results = client.lexical_search(
    collection_name=DEMO_COLLECTION,
    query=query,
    anns_field="vector_sparse",  # Sparse embedding field
    output_fields=["text", "category"],
    limit=5
)
```

**Advantages:**
- Fast and efficient
- Exact keyword matching
- Interpretable results

**Disadvantages:**
- No semantic understanding
- Misses synonyms and related concepts
- Sensitive to exact wording

### Hybrid Search

**Concept:**
Combines semantic and lexical search to get the best of both worlds. Uses both dense and sparse embeddings.

**How it Works:**
1. Perform semantic search (dense vectors)
2. Perform lexical search (sparse vectors)
3. Merge and re-rank results using a ranker

**Example:**
```python
Query: "machine learning embeddings"

# Semantic finds: ML concepts, neural networks, vector representations
# Lexical finds: exact matches for "machine learning" and "embeddings"
# Hybrid combines both for comprehensive results
```

**Implementation:**
```python
results = client.hybrid_search(
    collection_name=DEMO_COLLECTION,
    query=query,
    dense_field="vector_dense",
    sparse_field="vector_sparse",
    dense_weight=0.7,    # 70% semantic
    sparse_weight=0.3,   # 30% lexical
    ranker_type="weighted",
    limit=5
)
```

**Advantages:**
- Best recall (finds more relevant results)
- Balances semantic and exact matching
- Configurable weights for different use cases

**Disadvantages:**
- More complex to implement
- Requires tuning weights
- Slightly slower than single search


## 6. 📊 BM25 and Sparse Embeddings

### What is BM25?

**BM25** (Best Matching 25) is a probabilistic ranking function used in information retrieval. It's the gold standard for keyword-based search.

### BM25 Formula

```
BM25(D, Q) = Σ IDF(qᵢ) × (f(qᵢ, D) × (k₁ + 1)) / (f(qᵢ, D) + k₁ × (1 - b + b × |D| / avgdl))
```

Where:
- `D`: Document
- `Q`: Query
- `qᵢ`: Query term i
- `f(qᵢ, D)`: Term frequency in document
- `|D|`: Document length
- `avgdl`: Average document length
- `k₁`: Term frequency saturation (default: 1.5)
- `b`: Length normalization (default: 0.75)
- `IDF(qᵢ)`: Inverse document frequency

### Key Components

#### Term Frequency (TF)
How often a term appears in a document. More occurrences = higher relevance, but with diminishing returns.

#### Inverse Document Frequency (IDF)
How rare a term is across all documents. Rare terms are more informative.

```
IDF(term) = log((N - df + 0.5) / (df + 0.5))
```
- `N`: Total documents
- `df`: Documents containing term

#### Length Normalization
Prevents bias toward longer documents. Controlled by parameter `b`.

### BM25 in the Tutorial

The tutorial uses PyMilvus's `BM25EmbeddingFunction`:

```python
from pymilvus.model.sparse import BM25EmbeddingFunction

# Initialize BM25
bm25_ef = BM25EmbeddingFunction()

# Fit on corpus (learns IDF values)
bm25_ef.fit(corpus)

# Generate sparse embeddings
sparse_embeddings = bm25_ef.encode_documents(texts)
```

**Process:**
1. **Fit**: Analyze corpus to compute IDF values
2. **Encode**: Convert texts to sparse vectors
3. **Store**: Save sparse vectors in Milvus

### Sparse Vector Format

Sparse vectors store only non-zero values:

```python
# Dense vector (1024 dimensions):
[0.0, 0.0, 0.23, 0.0, 0.0, 0.45, 0.0, ...]  # Mostly zeros

# Sparse vector (same information):
{2: 0.23, 5: 0.45, 127: 0.89, ...}  # index: value
```

**Advantages:**
- Memory efficient (only store non-zero values)
- Fast computation (skip zero values)
- Interpretable (indices correspond to terms)

### Why Fit the Corpus?

BM25 needs to know:
1. **Document frequencies**: How many documents contain each term
2. **Average document length**: For normalization
3. **Vocabulary**: All unique terms in corpus

**Important:** The BM25 model must be fitted on the same corpus used for search. If you add new documents, you should refit the model.


## 7. 🔀 Hybrid Search and Ranking

### Ranking Strategies

When combining multiple search results, we need a strategy to merge and rank them. Milvus provides two main rankers:

#### Weighted Ranker

**Concept:**
Combine scores from different searches using weighted averages.

**Formula:**
```
final_score = w₁ × score₁ + w₂ × score₂
```

**Implementation:**
```python
from pymilvus import WeightedRanker

ranker = WeightedRanker(
    sparse_weight,  # Weight for lexical search
    dense_weight    # Weight for semantic search
)
```

**Example:**
```python
# Dense search score: 0.85
# Sparse search score: 0.60
# Weights: dense=0.7, sparse=0.3

final_score = 0.7 × 0.85 + 0.3 × 0.60 = 0.595 + 0.18 = 0.775
```

**When to Use:**
- You know the relative importance of each search type
- You want fine-grained control over ranking
- Different use cases need different weights

**Tuning Weights:**
- **High dense weight (0.7-0.9)**: Prioritize semantic similarity
- **Balanced (0.5-0.5)**: Equal importance
- **High sparse weight (0.7-0.9)**: Prioritize exact matches

#### RRF (Reciprocal Rank Fusion)

**Concept:**
Combine rankings from multiple searches without needing to normalize scores.

**Formula:**
```
RRF_score(d) = Σ 1 / (k + rank_i(d))
```

Where:
- `d`: Document
- `rank_i(d)`: Rank of document in search i
- `k`: Constant (typically 60)

**Implementation:**
```python
from pymilvus import RRFRanker

ranker = RRFRanker()
```

**Example:**
```python
# Document A:
# - Dense search rank: 1 (top result)
# - Sparse search rank: 5

RRF_score(A) = 1/(60+1) + 1/(60+5) = 0.0164 + 0.0154 = 0.0318

# Document B:
# - Dense search rank: 3
# - Sparse search rank: 2

RRF_score(B) = 1/(60+3) + 1/(60+2) = 0.0159 + 0.0161 = 0.0320

# Document B ranks higher (0.0320 > 0.0318)
```

**When to Use:**
- You don't want to tune weights
- Scores from different searches aren't comparable
- You want a robust, parameter-free method

**Advantages:**
- No parameter tuning needed
- Handles different score scales
- Proven effective in practice

**Disadvantages:**
- Less control than weighted ranking
- May not reflect domain-specific priorities

### Choosing a Ranker

| Scenario | Recommended Ranker | Configuration |
|----------|-------------------|---------------|
| Semantic-heavy (Q&A, RAG) | Weighted | dense=0.8, sparse=0.2 |
| Balanced (general search) | RRF or Weighted | dense=0.5, sparse=0.5 |
| Keyword-heavy (code search) | Weighted | dense=0.3, sparse=0.7 |
| Unknown requirements | RRF | Default |


## 8. 🏗️ Schema Design

### What is a Schema?

A **schema** defines the structure of a collection in Milvus. It specifies:
- Field names and types
- Primary key
- Vector dimensions
- Constraints

**Important:** Schemas are immutable after creation. Plan carefully!

### Field Types

#### Scalar Fields

**INT64:**
```python
FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True)
```
- Primary keys
- Timestamps
- Counters

**VARCHAR:**
```python
FieldSchema(name="text", dtype=DataType.VARCHAR, max_length=65535)
```
- Text content
- Categories
- Metadata strings

**FLOAT/DOUBLE:**
```python
FieldSchema(name="score", dtype=DataType.FLOAT)
```
- Numerical metadata
- Confidence scores

**BOOL:**
```python
FieldSchema(name="is_active", dtype=DataType.BOOL)
```
- Flags
- Binary states

#### Vector Fields

**FLOAT_VECTOR (Dense):**
```python
FieldSchema(name="vector_dense", dtype=DataType.FLOAT_VECTOR, dim=1024)
```
- Semantic embeddings
- Fixed dimensionality
- All values present

**SPARSE_FLOAT_VECTOR (Sparse):**
```python
FieldSchema(name="vector_sparse", dtype=DataType.SPARSE_FLOAT_VECTOR)
```
- BM25 embeddings
- Variable dimensionality
- Most values zero

### Schema Example from Tutorial

```python
fields = [
    # Primary key (auto-generated)
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
    
    # Text content
    FieldSchema(name="text", dtype=DataType.VARCHAR, max_length=65535),
    
    # Dense embedding (semantic)
    FieldSchema(name="vector_dense", dtype=DataType.FLOAT_VECTOR, dim=1024),
    
    # Sparse embedding (lexical)
    FieldSchema(name="vector_sparse", dtype=DataType.SPARSE_FLOAT_VECTOR),
    
    # Metadata for filtering
    FieldSchema(name="category", dtype=DataType.VARCHAR, max_length=100),
    FieldSchema(name="source", dtype=DataType.VARCHAR, max_length=100),
    FieldSchema(name="timestamp", dtype=DataType.INT64)
]

schema = CollectionSchema(
    fields=fields,
    description="Demo collection for tutorial"
)
```

### Design Considerations

#### 1. Primary Key
- Must be unique
- Can be auto-generated or user-provided
- Used for updates and deletions

#### 2. Vector Dimensions
- Must match embedding model output
- Cannot be changed after creation
- Common sizes: 384, 768, 1024, 1536

#### 3. VARCHAR Length
- Set `max_length` appropriately
- Larger = more storage
- Maximum: 65,535 characters

#### 4. Metadata Fields
- Add fields you'll filter on
- Keep it minimal (affects performance)
- Consider cardinality (unique values)

### Best Practices

**DO:**
- Plan schema before creation
- Include fields for common filters
- Use appropriate data types
- Document your schema

**DON'T:**
- Add unnecessary fields
- Use VARCHAR for numerical data
- Forget about future requirements
- Change schema after creation (requires recreation)


## 9. 🎯 Singleton Pattern

### What is the Singleton Pattern?

The **Singleton pattern** ensures a class has only one instance throughout the application lifecycle. It's used in the tutorial for managing connections and resources.

### Why Use Singleton?

**Problems it Solves:**
1. **Resource Management**: Avoid multiple database connections
2. **State Consistency**: Share state across application
3. **Performance**: Reuse expensive objects (embeddings, connections)

### Implementation in Tutorial

```python
# From utils/singleton.py
class Singleton(type):
    _instances = {}
    
    def __call__(cls, *args, **kwargs):
        if cls not in cls._instances:
            cls._instances[cls] = super(Singleton, cls).__call__(*args, **kwargs)
        return cls._instances[cls]
```

### Usage

```python
class MilvusClient(metaclass=Singleton):
    def __init__(self, config, embeddingsconfig):
        self.client = MC(uri=..., secure=True)
        self.embeddings = EmbeddingClient(embeddingsconfig)
        self.sparse_embeddings = SparseEmbeddingClient()
```

**Result:**
```python
# First call: Creates instance
client1 = MilvusClient(config, embeddings_config)

# Second call: Returns same instance
client2 = MilvusClient(config, embeddings_config)

# client1 is client2 == True
```

### Benefits in Tutorial

1. **Connection Pooling**: Single Milvus connection reused
2. **Embedding Cache**: Embedding clients initialized once
3. **BM25 Model**: Fitted model shared across operations
4. **Memory Efficiency**: No duplicate resources

### When to Use Singleton

**Good Use Cases:**
- Database connections
- Configuration managers
- Logging systems
- Resource pools

**Avoid When:**
- Testing (hard to mock)
- Multi-threading (needs thread safety)
- Multiple configurations needed


## 10. ⚡ Performance Optimization

### Batch Operations

**Concept:**
Process multiple items together instead of one at a time.

**Why it Matters:**
- Reduces network overhead
- Amortizes connection costs
- Enables parallel processing

**Example from Tutorial:**
```python
batch_size = 10
for i in range(0, len(documents), batch_size):
    batch = documents[i:i + batch_size]
    client.insert_data(collection_name=DEMO_COLLECTION, documents=batch)
```

**Recommended Batch Sizes:**
- **Insertion**: 1,000 - 5,000 records
- **Embedding generation**: 100 - 500 texts
- **Search**: 10 - 100 queries

### Index Selection Strategy

**Decision Tree:**

```
Is dataset < 10,000 vectors?
├─ Yes → Use FLAT (exact search)
└─ No → Continue

Is low latency critical?
├─ Yes → Use HNSW (fast, high memory)
└─ No → Continue

Is memory limited?
├─ Yes → Use IVF_SQ8 (compressed)
└─ No → Use IVF_FLAT (balanced)
```

### Field Selection

**Concept:**
Only retrieve fields you need.

**Bad:**
```python
results = client.query(
    collection_name=DEMO_COLLECTION,
    output_fields=["*"]  # Returns ALL fields
)
```

**Good:**
```python
results = client.query(
    collection_name=DEMO_COLLECTION,
    output_fields=["text", "category"]  # Only what you need
)
```

**Impact:**
- Reduces network transfer
- Faster serialization
- Lower memory usage

### Filter Optimization

**Concept:**
Apply filters before vector search when possible.

**Strategy 1: Pre-filtering**
```python
# Filter first, then search
results = client.semantic_search(
    collection_name=DEMO_COLLECTION,
    query=query,
    filter='category == "tutorial"',  # Reduces search space
    limit=10
)
```

**Strategy 2: Post-filtering**
```python
# Search first, filter results
results = client.semantic_search(
    collection_name=DEMO_COLLECTION,
    query=query,
    limit=100  # Get more results
)
# Then filter in application code
```

**When to Pre-filter:**
- Filter is highly selective (< 10% of data)
- Filter field is indexed
- Exact matches needed

**When to Post-filter:**
- Filter is not selective (> 50% of data)
- Complex filter logic
- Approximate results acceptable

### Connection Management

**Best Practices:**

1. **Reuse Connections:**
```python
# Good: Single client instance (Singleton)
client = MilvusClient(config, embeddings_config)

# Bad: New connection each time
def search(query):
    client = MilvusClient(config, embeddings_config)  # Don't do this!
    return client.search(...)
```

2. **Load Collections:**
```python
# Load collection into memory before searching
client.load_collection(DEMO_COLLECTION)

# Now searches are fast
results = client.semantic_search(...)
```

3. **Flush After Bulk Inserts:**
```python
# Insert data
client.insert_data(collection_name=DEMO_COLLECTION, documents=batch)

# Flush to ensure persistence
client.flush_collection(DEMO_COLLECTION)
```

### Memory Management

**Strategies:**

1. **Vector Quantization:**
   - Use IVF_SQ8 instead of IVF_FLAT
   - Reduces memory by ~75%
   - Slight accuracy loss (1-5%)

2. **Dimension Reduction:**
   - Use smaller embedding models
   - 384-dim instead of 1024-dim
   - Trade-off: Lower accuracy

3. **Partitioning:**
   - Split large collections into partitions
   - Search only relevant partitions
   - Reduces memory footprint

### Monitoring and Tuning

**Key Metrics:**

1. **Search Latency:**
   - Target: < 100ms for most applications
   - Tune: Index parameters, batch size

2. **Recall:**
   - Target: > 95% for most applications
   - Tune: Index type, search parameters

3. **Memory Usage:**
   - Monitor: Collection size, index overhead
   - Tune: Quantization, partitioning

4. **Throughput:**
   - Target: Application-specific
   - Tune: Batch size, connection pooling
